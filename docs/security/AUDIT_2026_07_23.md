## 2. Kanonski popis uloga (žive baze, provjereno)

### Projekti (`project_members.role` — `text`, ne enum)
- **owner** — virtualno, `projects.user_id = auth.uid()`; pun pristup.
- **member** — pun pristup u dopuštenim domenama (transakcije/milestone-i unutar projekta, ovisno o `is_project_member` helperu).
- **worker** — smije bilježiti SVOJE work_entries; ne dobiva financijski digest (v. `enqueue_participant_digest_event`); ne smije write nad financijskim entitetima.
- **investor** — read scope u frontendu (InvestorPhasesView); po dizajnu **ne smije** vidjeti expenses/worker_payouts/work_entries/funding/documents/milestone.total_budget.
- **viewer** — nije prisutan u živoj bazi. Kod referencira samo `member|worker|investor`. Preskočeno; defenzivno pokriveno u §3 tvrdnjom da RLS ne pretpostavlja `viewer=member`.

### Krug (`krug_membership.role`)
- **owner** — `krug.owner_id`, virtualno.
- **punopravni** — pun member; upravljanje shared paymentima, glasanje o brisanjima, dodavanje članova.
- **obicni** — pasivni sudionik; smije vidjeti krug i sudjelovati u proposed/act flowu, **ne smije** promocija sebe, brisanje drugih, dodavanje shared sourcea, promjena meta.

### Biznis / kolaboracija
- **owner** — `business_profiles.user_id`.
- **collaborator** — `project_collaborators` (Limited Access / Full Access flag); RLS gate preko `has_project_permission`.

### Admin sloj
- **admin** — `user_roles.role='admin'`, provjerava se preko `has_role(uid, 'admin')` (SECURITY DEFINER, safe od RLS rekurzije).

## 4. SECURITY DEFINER funkcije — provjera guarda (izvod)

Sve inspicirane funkcije derivirano vežu identitet na `auth.uid()` **prije** dodira podataka:

- `soft_delete_record(text,uuid)` — `raise if auth.uid() is null`, whitelist tablica, dinamički `UPDATE ... WHERE user_id = $1 OR is_admin`.  **✔ safe**
- `align_source_to_bank(uuid, numeric, timestamptz)` — `if v_owner <> v_caller then raise 42501`. **✔ safe**
- `set_source_anchor(...)` — analogno; owner check.
- `create_worker_payout(...)` — `SELECT projects.user_id FOR UPDATE; if v_owner_id <> v_caller raise`. **✔ safe**
- `grant_module_access(...)` — `v_admin := auth.uid()` + poziv se zaključava `has_role(v_admin, 'admin')` iznutra prije INSERT-a (v. definicija). **✔ safe**
- `activate_module_trial(_module)` — potpis nema `_user_id` parametra; radi se strogo `auth.uid()`.  **✔ safe**
- `find_user_by_email(_email)` — vraća samo `id`, ne cure PII izvan onoga što je nužno za invitations flow.
- `enqueue_participant_digest_event(...)` — filter `role='worker'` (nedavni fix); pokriveno vitest testom.
- `has_role`, `has_entitlement`, `is_project_member`, `is_krug_member`, `is_projects_subscriber`, `can_write_module` — read-only pomoćnici; svi rade s eksplicitnim `_user_id` parametrom pozivanog iz policy contexta gdje se prosljeđuje `auth.uid()`.

**Nalaz statičke analize:** nema funkcije koja piše bez ownership/admin gate-a. Svi write RPC-ovi eksplicitno usporedjuju `auth.uid()` s vlasnikom projekta/sourcea/kruga.

## 5. Edge funkcije — auth guard (izvod)

- `financial-assistant/index.ts` — `auth.getClaims(token)` iz `Authorization` headera → `userId` iz `claims.sub`; sve DB upite drži scoped na `userId`. **✔**
- `paddle-webhook/index.ts` — HMAC-SHA256 provjera potpisa; user identifikacija isključivo iz `customData.user_id` unutar potpisom verificiranog eventa. **✔**
- `paddle-portal-url/index.ts` — čita `auth.getUser(token)`, dohvaća `customer_id` iz `user_entitlements.metadata` za taj `user_id`; ne prihvaća user_id iz bodyja.
- `parse-receipt`, `bank-sync-transactions`, `import-statement-*` — svi počinju s `getUser()`; ako nema JWT → 401.
- Admin funkcije (`admin-manage-user`, `list-users`, `admin-grant-*`) — provjera `has_role(uid, 'admin')` prije bilo koje akcije.

**Nalaz:** nema edge fn-a koji izvodi identitet iz bodyja korisničkog zahtjeva.

## 6. Data-API GRANT sanity

Sve žive tablice imaju eksplicitne GRANT-ove (`authenticated` + `service_role`; `anon` samo tamo gdje postoji permissive read policy — landing/plans/publikacije). Poništene su ranije implicitne pretpostavke iz remixa.

## 7. "Crveni kandidati" koje trajni test paket MORA dokazati u produkciji

Statička analiza nije dovoljna za sljedeće — potrebno je pokrenuti `npm run test:security`:

1. **investor + `is_project_member` helper:** helper vraća `true` za sve project_members retke bez razlike po ulozi. Ako neka RLS policy nad `expenses` / `project_funding` / `project_worker_payouts` koristi taj helper bez dodatnog `role = 'owner'/'member'` filtera, investor bi mogao čitati financije. Spec `03-investor-scope` to eksplicitno provjerava. **Ako ovi testovi padnu → KRITIČNO.**
2. **`project_documents` po ulozi:** analogno — dokumenti mogu sadržavati ugovorne iznose. Test `investor NE SMIJE čitati project_documents` je gate.
3. **`preview_source_balance_after_batch`** — nije 100% jasno da funkcija filtrira po vlasniku sourcea; spec pokriva pokušaj tuđim source_id.
4. **`paddle-portal-url`** — spoofanje `user_id` u bodyju; spec tvrdi da response ne sadrži A-in id.
5. **`profiles.email`** — cross-user leak; spec provjerava da B ne vidi A-in email.
